The core of the stateful simulation is the “ping-pong” technique. This method uses two Frame Buffer Objects (FBOs)—off-screen rendering targets—to iteratively update the simulation state. In each frame, a shader program reads the current state of all entities from an input texture, calculates the new state, and writes the results to an output texture. For the subsequent frame, the roles of the input and output textures are swapped. The system uses Transform Feedback, a core WebGL feature, to capture the outputs of the vertex shader directly into a GPU buffer, enabling the updated particle states to be used as input for the next simulation step. This cycle allows the simulation to evolve over time entirely on the GPU. In this GPGPU context, textures are not used as visual images but as 2D data arrays—a paradigm often called “data-as-texture.” The four RGBA channels of each pixel store numerical data, with each pixel corresponding to a unique entity in the simulation. To prevent numerical instability and quantization errors that can accumulate and corrupt the simulation, it is critical to use high-precision floating-point texture formats, such as RGBA32F, instead of standard 8-bit integer formats. While the simulation runs efficiently on the GPU, its state must eventually be read back to the CPU to drive DOM updates. The primary performance challenge in this process is the readPixels bottleneck. The standard WebGL readPixels function is a synchronous, blocking operation. When called, it stalls the JavaScript main thread until the GPU has finished all pending commands and the data has been copied, completely negating the performance benefits of GPU acceleration. The definitive solution implemented in this architecture is the Asynchronous GPU Data Readback Pattern (AGDRP). This is achieved using Pixel Buffer Objects (PBOs), a hardware-level optimization in WebGL2 that enables non-blocking data transfers. A PBO allows the CPU to initiate a data transfer from the GPU and continue with other work without waiting. The transfer happens in the background, managed by the GPU driver, eliminating the pipeline stall. This strategy represents a conscious architectural trade-off: we accept a few frames of latency (\~50ms) in the data arriving at the CPU in exchange for a completely non-blocking main thread. A related software-level orchestration pattern can be implemented using fenceSync and non-blocking clientWaitSync checks to achieve a similar latency-hiding effect. This asynchronous pipeline delivers a continuous stream of high-frequency data to the CPU, which is then consumed and managed by the system’s reactive control plane. 4.0 The Reactive Control & Presentation Plane The reactive dataflow graph acts as the system’s declarative “nervous system.” Implemented with reactive stream libraries, it is responsible for managing asynchrony, orchestrating control signals from user input, and transforming the raw data from the GPGPU engine into its final rendered state. This declarative approach provides a clear, predictable, and manageable pipeline for the entire application’s data lifecycle. The architecture primarily leverages three constructs from the reactive streams paradigm: Streams: These are the fundamental conduits of the system, representing sources of values delivered over time. They can represent high-frequency data arriving from the GPU simulation, low-frequency user gestures like mouse scrolls, or configuration changes from UI elements. Transducers: These are composable, memory-efficient data transformation functions. Chained together, they process data as it flows through a stream without creating intermediate collections. For example, a transducer is used to map a raw noise value from the GPGPU’s \[-1, 1\] range to a valid CSS font weight in the \[100, 900\] range. Subscriptions: These are the consumers at the end of a data stream. They are responsible for executing side effects, such as taking the final, processed data and surgically updating a specific DOM element’s style property. These individual streams are composed into a declarative dataflow graph using @thi.ng/rstream-graph. This graph serves as a “Synchronization Crucible,” where data streams operating on different timescales are merged. For example, the high-frequency physics data arriving from the GPU must be combined with low-frequency user inputs from UI sliders. Operators like sync() ensure that a complete and consistent state is emitted only when all constituent input streams have provided a new value, preventing visual artifacts and maintaining application stability. Instead of comparing entire tree structures to find changes—an O(n) operation—reactive subscriptions are attached directly to the target DOM elements. These subscriptions perform a surgical update by directly modifying the style.fontVariationSettings property.
